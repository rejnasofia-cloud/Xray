What each role needs to do

PM

Owns onboarding timeline & sign-off
  • Raise Jira project setup request using CoE template
  • Assign QA lead and dev lead to Xray roles in project settings
  • Set onboarding completion milestone in the project plan
  • Review and sign off on the coverage dashboard
  • Confirm all 5 CoE metrics are being tracked and visible
QA

Leads test asset creation & execution
  • Create test repository folder structure in Xray
  • Write or import all manual test cases
  • Define BDD/Gherkin scenarios and link feature files
  • Create test plans and test executions per sprint
  • Link all test cases to user stories / requirements
  • Configure CI integration for automated test results
  • Set up traceability matrix and requirements coverage report

Owns automation pipeline integration
  • Set up CI/CD pipeline to push results to Xray

Embeds Xray into sprint ceremonies
  • Add Xray test plan creation to sprint planning checklist

  • Add coverage report review to definition of done

Choose your migration path

  1. 1
    Export all test cases from Zephyr Scale or Squad using the built-in CSV export (Tests > Export). Include fields: name, steps, expected result, labels, priority.
  2. 2
    Clean the CSV: normalise step format to Xray's Action | Data | Expected Result columns. Remove Zephyr-specific fields such as cycle ID and version ID.
  3. 3
    In Xray Cloud, go to the Test issue type > Import > CSV. Map columns to Xray fields in the mapping UI. Run a test import on 10 rows first before doing the full batch.
  4. 4
    Re-create test sets and test plans in Xray. Zephyr cycles map to Xray Test Plans; Zephyr test sets map to Xray Test Sets.
  5. 5
    Re-link migrated tests to Jira stories using the "Tests" panel on each story, or bulk-link via the Xray import API for large volumes.
  6. 6
    Validate coverage by opening the Requirements Coverage gadget on your Jira dashboard and confirming all requirement-to-test links are intact.
⚠ Zephyr execution history does not migrate. Archive your Zephyr project for historical reference but do not attempt to import past execution runs into Xray.
  1. 1
    Standardise your spreadsheet to four columns: Test Name | Step Action | Step Data | Expected Result. Each row is one step; repeat the test name across rows for multi-step tests.
  2. 2
    Add a Labels column for test type tags: manual, regression, smoke, exploratory. These map directly to Xray labels on import.
  3. 3
    Save as UTF-8 CSV. Remove merged cells, colour formatting, and any formula-driven cells before exporting.
  4. 4
    Import via the Xray Cloud CSV importer. Select "Create Test issues" and map your columns to Xray fields in the mapping interface.
  5. 5
    Assign imported tests to Test Repository folders matching your feature areas (for example: /Authentication, /Checkout, /Reporting).
  6. 6
    Link tests to epics or stories manually using the Tests panel on each story, or use Xray's bulk link feature under Test Plans.
✓ If your sheet has more than 200 tests, split by feature area and import in batches. This makes it easier to catch and fix column-mapping errors early.
  1. 1
    Start by mapping your Jira epics and stories. Every story with acceptance criteria is a candidate for test coverage. Export your backlog to get the full picture before writing tests.
  2. 2
    Create a Test Repository structure in Xray that mirrors your application modules (e.g. /Module A, /Module B). This becomes the permanent home for all test cases across releases.
  3. 3
    Write manual test cases using the Test issue type in Xray. Aim for one test case per acceptance criterion on your current sprint stories to start.
  4. 4
    For BDD teams: write Gherkin scenarios in Xray as Cucumber-type tests. Use the Xray <> Git integration to sync .feature files to your repository automatically.
  5. 5
    Create your first Test Plan for the current sprint. Add the relevant test cases. This generates your first Test Execution record and begins tracking run history.
  6. 6
    Link test cases to user stories via the "Tests" panel on each story. Run your first execution and verify the Coverage gadget reflects test status on the story.
✓ Don't try to write tests for the entire backlog upfront. Start with the current sprint and the top 3 highest-risk features, then expand coverage progressively each sprint.

CoE tracked metrics — all 5 must be live before sign-off

Test execution pass rate
—
% of test runs marked PASS in Xray executions
Requirements coverage
—
% of stories with at least 1 linked test case
Defect density
—
Defects linked to test executions per sprint
Test plan completion
—
% of test cases executed within the sprint plan
Automation coverage
—
% of test cases backed by automated execution

How to surface each metric in Jira

Pass/fail rate Xray Test Execution Results gadget → filter by project + sprint
Requirements coverage Xray Requirements Coverage gadget → set fix version or sprint label
Defect density Jira filter: issue type = Bug, linked to Test Execution, grouped by sprint
Test plan completion Test Plan Board gadget → view % executed vs total tests in the plan
Automation coverage Xray filter: Test type = Generic or Cucumber, execution status = PASS/FAIL
✓ CoE sign-off criteria: All 4 onboarding phases complete + all 5 metric gadgets live on a shared Jira dashboard = project marked as adopted in the CoE tracker.
